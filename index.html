Squatch Food Waste Application

Client/Developer Documentation      Version 1.0     June 04, 2020

Serena Bowen, Ashlyn Hartnett, Kadyn Marshall

Motivating Summary

Sustainable Connections directs a food recovery program, known as the Squatch Waste Campaign, that manages food donations from restaurants, hotels, and banquets as well as individual drivers or driver organizations who deliver recovered items to recipient agencies which then distribute the food to those in need. The donors supply food to the program that would otherwise be thrown out, which adds up to tens of thousands of meals for those in need per year and nets the donors lessened sanitary service fees and a tax break for diverting waste. The program conforms to food safety laws and requires that the food donors do the same. The drivers at Sustainable Connections must keep track of how much food is being donated, and from which organizations, to ensure that they continue to receive grants from the Washington State Department of Ecology in the future.

Historically, the program admins have been frustrated by low data retention, lack of reporting, non-standardized logging, and loss of data collected on physical forms. The food recovery drivers would often collect data on paper or personal apps which then had to be standardized and digitized at a later time.

Product Summary

To this end, the Squatch Food Waste apps were designed to be a software solution for data collection, aggregation, and reporting. The web and mobile applications allow admins to track safety, type, and quantity of food donations, create and revise routes (e.g., lists of donor organizations scheduled to be visited at set intervals), track the amount and type of food delivered to recipient organizations, and track the amount and type of food recovered from donor organizations. The method of food data collection is through manual data entry into the mobile application and the method of food data aggregation is through report creation using the web application.

Project Technical Details

Database

The food recovery app uses a MongoDB database. MongoDB is a no-sql database, which works really well with JavaScript applications because it allows objects to be stored in a very similar format to how they appear in JavaScript code. This is helpful because the food-recovery codebase is written almost entirely in JavaScript.

The data is stored using documents and collections, as opposed to rows and tables. The collections are where the documents are stored, and the documents themselves are similar to objects. With the food recovery application, there are separate collections for users, organizations, route-templates, etc. By categorizing the collections, we can easily save and query JavaScript objects into and out of the database from their respective collections. In this way, the database acts as a location to store objects.

Database Structure:

There are collections for users, organizations, route-templates, route-instances, random-donations, and daily messages.

  • Users: The users collection stores documents containing information pertaining to individual users, including their name, contact info, admin status, etc.
  • Organizations: The organization collection includes information for both the donor and recipient organizations.
  • Route Templates/Route Instances: Route templates hold the data including the stops of a route, which days that it should occur on, and other scheduling information. These templates are then used by the server to create route instances, which is what gets updated with the actual donation information that is filled out by the users.
  • Random Donations: Random donations are for storing data regarding one-time food recovery pickups. They have not been used yet because the functionality for random donations has not been implemented at this time.
  • Messages: The message collection stores the text of each message and the start and end date of when to show the message on the Home screen.

To see all the details of each schema, see the models directory on the backend repository.

Heroku

Using git to push to remote staging apps (backend, website)

Both the server (backend) and the website have staging versions on Heroku to allow us to modify code and experiment without disrupting service for the main Heroku app. Heroku calls this config a pipeline. Code changes can be tested using the staging app then deployed to the main app with only one click on the Heroku pipeline page (on the Heroku website). Changes made to the staging app will have no effect on the main app as long as the server URL (for the website), the database url (for the server), and the AWS Cognito configuration (for both) are correctly set. These are set as constants in the app config files.

To set up a git connection to the staging app locally, you'll want to execute the following commands in the same directory as your git repository for the same app:

heroku login

heroku apps (optional, to confirm the apps are shown correctly)

heroku git:remote -a [name of heroku staging app]

git remote rename heroku heroku-staging (give our new Heroku remote a sensible name - Heroku CLI names all of the Heroku remotes it creates for your app "heroku", so we don't want to overwrite the deployed remote when we add our new remote [source])

Repeat the above two steps for the deployed versions of the app if desired.

From here on, if you want to push to a heroku app, you'll specify which one to push to by entering:

git push [name of heroku remote] [name of local branch to push up], e.g. git push heroku-deployed user_testing (pushes up stable version of backend for use in user testing)

Misc. helpful commands:

  • git fetch --all (fetch info on all branches on all remotes)
  • git branch -a (show all branches locally in green, all remote branches in red)
  • vim .git/config (view/ edit git config, useful if you know what changes you want to make to the branches/ remotes but don't know the git command)
  • heroku apps -h (show commands you can use to manipulate Heroku apps)

Running the App

How to install npm
  1. The easiest way to install node and npm is by installing the Node Version Manager (NVM). Follow these instructions to do so: https://github.com/nvm-sh/nvm
  2. Once NVM is installed, you can install node by specifying a version. Run the command ‘nvm install 12.0.0’ or to install the latest version run ‘nvm install node’
How to run the app
  • Once you have npm and node installed, navigate to the mobile folder inside the main foodrecovery folder.
  • Before running the application, you must make sure all the necessary packages will be available for the application. In the command line run, ‘npm install’ to install all imports. Whenever you add a new package or import to the application run ‘npm install’ to reinstall all packages or ‘npm install package-name’ to just install the specific package.
  • After you all packages are installed, type in ‘npm start’ to start the bundler.
How to connect an emulator
  1. To run on an Android emulator, first install Android Studio and set up a Virtual Device. Here are some instructions on how to do that: https://developer.android.com/studio/run/managing-avds.
  2. Open Android Studio and click on Configure in the bottom right corner. Then click on AVD Manager.

  1. Select the Virtual Device you want to use and click on the green Play button to start the emulator.

  1. Now that the emulator is running, go back to the terminal where the app bundler is running and type ‘a’ to open the app on the emulator.

How to run on an external device
  • To run the application on an external device, look to the instructions on installing the Expo app in the section ‘Expo Information’.
  • After you install and sign into the Expo app and the app bundler is running, it should show up under the section “Recently in Development”. Click on the application running and it should start to build the JavaScript bundle. The progress of building the bundle will show in the terminal where you have the app bundler running.
  • If the app doesn’t show up in the list of ‘Recently in Development’ or if you get the error saying ‘Packager not running’, check to make sure both your computer and phone are connected to the same Wifi.
  • Note: this will only work if you are on a Private network. Unfortunately, we haven’t found a way to run the app on Public networks (WWU on-campus Wifi).

Cognito Information

How to sign in, view User Pool information

To access the user accounts managed by Cognito (used to provide password management, provide access tokens to privileged endpoints on the server), you’ll first need to sign into the AWS console (https://console.aws.amazon.com/) and navigate to the Cognito service homepage (https://us-west-2.console.aws.amazon.com/cognito/home?region=us-west-2).

How to manage user pools

From the Cognito homepage, you’ll select the option to “Manage User Pools”

then select the user pool corresponding to the version of the app whose users you want to view (either staging version or production version).

Once selected, you can open the Users and groups submenu to view the Cognito username, account status, email verification status, and creation and update times for all user accounts associated with the selected app. It will also give you the option to create new Cognito users non-programmatically.

Clicking on a specific user will give you the option to reset their password, view their attributes, view their sign-in history, and provide the option to delete said user (click “Disable User” -> click “Delete User”).

Opening the Attributes submenu will allow you to view the required attributes for User Pool members and the permissible sign-in methods.

Opening the Policies submenu will allow you to view/ edit the password requirements, user creation settings, and the duration of temporary password validity.

Opening the Message Customizations submenu will allow you to view/ edit the messages users receive when a) invited to use the app and b) asked to verify their email.

Opening the App Clients submenu will allow you to view the App client ids for the web and mobile apps. These ids are set in the aws-exports.js file for the web and mobile to allow the Amplify library to perform simple user actions such as sign-in or change own password. The server does not have an app client associated.

How to manage IAM app users

To access the IAM accounts which provide/ control access to AWS resources (used to provide access to the AWS console/ services to collaborating developers, provide access keys to apps using AWS-SDK), you’ll first need to sign into the AWS console (https://console.aws.amazon.com/) and navigate to the IAM service homepage (https://console.aws.amazon.com/iam/home?region=us-west-2#/home).

Each app which uses AWS-SDK should have its own account with the Cognito User Pools policy attached. This policy can be directly attached to each app IAM account or attached to a group with each app IAM account as a member.

All developer IAM accounts should be at least have the AmazonCognitoPowerUser policy attached. Again, this policy can be directly attached to each developer IAM account or attached to a group with each app IAM account as a member.

To learn more about IAM account management, refer to the feature guide here: https://aws.amazon.com/iam/features/manage-users/ 

How to check config vars

Now that we have the app clients, user pools, and app IAM accounts set up, we need to make sure our mobile app, web app, and server have the correct Cognito config variables attached.

The web and mobile app require the region, user pool id, and user pool app client id to be set to use the Amplify library and require the IAM app user credentials to be set to use the AWS-SDK. These variables are set in the aws-exports.js files.

The user pool id and user pool app client id have two possible values for each mobile and web app: staging setting and production setting. This is because we use different user pools for staging and for production. When operating in the one setting, you’ll want to uncomment the configuration corresponding to the context you’re using and comment the configuration corresponding to the opposite context. The IAM app user credentials will remain the same in both configs because it has access to all Cognito user pools.

The server requires only the region and the user pool id, and these are set in server.js inside the cognitoExpress constructor. Please update the user pool id to correspond to the Heroku app version (production or staging) the server code will be running on.
